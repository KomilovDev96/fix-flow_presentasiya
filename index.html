Обзор системы

Назначение, цели, охват и ключевые характеристики продукта

FixFlow — это веб-ориентированная платформа класса Service Desk, разработанная для автоматизации процессов управления заявками на ремонт оборудования, технического обслуживания помещений, ИТ-поддержки и других видов сервисных запросов внутри крупных организаций с разветвлённой филиальной структурой.

🌟
Цель проекта
  • Сокращение времени реакции на инциденты
  • Централизованный учёт всех сервисных запросов
  • Повышение прозрачности выполнения работ
  • Контроль SLA и KPI исполнителей
🎯
Целевая аудитория
  • Торговые сети и ритейлеры
  • Банки и финансовые учреждения
  • Производственные предприятия
  • Управляющие компании (недвижимость)
⚡
Ключевые ценности
  • Мобильный доступ через Telegram Mini App
  • Real-time уведомления (WebSocket)
  • Dashboard-аналитика в стиле BI-систем
  • Полный audit log всех действий

Супер Администратор

Полный контроль над системой: пользователи, организации, конфигурации, безопасность

🛡
Роль SUPER_ADMIN — максимальный уровень привилегий
Супер Администратор — это системная роль, недоступная через обычную регистрацию. Назначается вручную на уровне БД. Имеет доступ ко всем организациям, настройкам системы, логам безопасности и инфраструктурным конфигурациям. Действия SUPER_ADMIN логируются с повышенным приоритетом в audit log.
🏠 Управление организациями
  • Создание новых тенантов (организаций)
  • Настройка лимитов: пользователи, заявки, хранилище
  • Блокировка / разблокировка организации
  • Просмотр статистики по каждому тенанту
  • Настройка тарифного плана
👤 Управление пользователями
  • Просмотр всех пользователей системы
  • Создание / удаление / блокировка аккаунтов
  • Принудительный сброс пароля
  • Изменение роли любого пользователя
  • Просмотр сессий и активных устройств
  • Инвалидация всех токенов пользователя
📊 Глобальная аналитика
  • Дашборд по всем организациям сразу
  • Сравнение KPI между тенантами
  • Топ-активных объектов и исполнителей
  • Отчёты по нагрузке на систему
  • Метрики использования: API calls, storage, WS
⚙ Системные настройки
  • Настройка глобальных SLA-конфигураций
  • Управление справочниками (группы, категории) глобально
  • Настройка Telegram Bot (webhook, токен)
  • Управление S3/MinIO хранилищем
  • Конфигурация уведомлений (шаблоны, каналы)
📄 Аудит и безопасность
  • Полный audit log всей системы
  • Фильтрация по пользователю, IP, действию, дате
  • Экспорт логов за любой период
  • Просмотр всех активных JWT-сессий
  • Принудительный logout всех пользователей
  • Просмотр попыток взлома / brute-force
🚀 Инфраструктура
  • Просмотр состояния очередей BullMQ
  • Управление failed/stalled jobs
  • Мониторинг Redis (память, ключи, TTL)
  • Просмотр ошибок из Sentry прямо в UI
  • Принудительный запуск cron-задач

SLA-контроль и мониторинг

Уровни сервиса, эскалации и автоматические триггеры

ℹ
Что такое SLA в FixFlow
SLA (Service Level Agreement) — это максимально допустимое время от создания заявки до её закрытия, определяемое приоритетом заявки. Система автоматически отслеживает дедлайны через BullMQ-очередь и отправляет эскалационные уведомления.

Таблица SLA по приоритетам

ПриоритетВремя реакции (APPROVED)Время выполненияЭскалация 1Эскалация 2
CRITICAL 15 минут 1 час 30 мин → уведомление менеджеру 1 ч → уведомление руководителю
HIGH 1 час 4 часа 2 ч → напоминание 4 ч → эскалация менеджеру
MEDIUM 4 часа 2 рабочих дня 1 день → напоминание 2 дня → эскалация
LOW 1 рабочий день 5 рабочих дней 3 дня → напоминание 5 дней → эскалация

Техническая реализация SLA

⏰
BullMQ Job Queue
  • При создании заявки создаётся delayed job в Redis
  • Триггер эскалации 1: задержка = SLA/2
  • Триггер эскалации 2: задержка = SLA
  • При закрытии заявки jobs удаляются
  • Cron-задача: ежечасная проверка просроченных
📊
SLA Метрики
  • sla_deadline — рассчитывается при создании
  • sla_status: ON_TIME / AT_RISK / BREACHED
  • AT_RISK: осталось менее 20% от SLA-времени
  • Цветовая индикация в таблице заявок
  • SLA-метрики в analytics dashboard

Производительность Admin-панели

Целевые метрики скорости, оптимизации и UX-требования

⚡
Цель: Admin-панель должна работать мгновенно
Администраторы и менеджеры работают в панели весь рабочий день. Каждые 100ms задержки = потеря производительности. Целевые показатели спроектированы под high-frequency usage: фильтрация, сортировка, переход между разделами — всё должно быть мгновенным.

Технические методы достижения скорости

⚡
Frontend оптимизации
  • Code splitting — каждый раздел загружается lazy (React.lazy + Suspense)
  • React Query — кеширование API-ответов на клиенте, stale-while-revalidate стратегия
  • Virtual scroll (react-virtual) — таблицы с 10 000+ строк без лагов
  • Debounce на всех фильтрах — 300ms задержка перед API-запросом
  • Оптимистичный UI — статус заявки меняется мгновенно, подтверждается от сервера
  • Vite bundler — сборка за < 10с, HMR за < 200мс в dev
  • Gzip / Brotli сжатие на Nginx — JS бандл < 250 KB (gzip)
💾
Backend оптимизации
  • PostgreSQL индексы — по status, priority, created_at, executor_id, object_id
  • Redis кеш — аналитика и справочники кешируются на 5 минут
  • Server-side пагинация — возвращается максимум 50 записей за запрос
  • Составные индексы — для частых фильтров (status + priority, date + object)
  • Connection pooling (pg-pool: min=5, max=20) — без overhead на подключение
  • Partial indexes — отдельный индекс по активным заявкам (WHERE closed_at IS NULL)
  • Query explain analyze — все сложные запросы проверяются в Seq Scan → Index Scan

Почему именно эти технологии?

Обоснование выбора каждого компонента стека — факты, не маркетинг

NestJS Backend Framework
✓Модульная архитектура — каждый модуль (Requests, Users, SLA) изолирован, легко тестируется и переиспользуется без циклических зависимостей
✓TypeScript из коробки — полная типизация DTO, entities, responses. Ошибки типов ловятся на этапе компиляции, а не в production
✓Dependency Injection — сервисы внедряются автоматически, код тестируется с mock-провайдерами без сложных патчей
✓WebSocket Gateway + BullMQ — нативная поддержка через пакеты @nestjs/websockets и @nestjs/bull, конфигурируется в 10 строк
✓Guards / Interceptors / Pipes — RBAC, валидация и логирование подключаются декораторами, не захламляя бизнес-логику
Альтернатива: Express.js — потребует ручной настройки архитектуры, нет DI, TypeScript нужно настраивать отдельно. Fastify — быстрее, но экосистема меньше, меньше готовых решений для enterprise.
PostgreSQL Primary Database
✓ACID транзакции — смена статуса заявки, создание assignment и отправка уведомления атомарно в одной транзакции. Никаких inconsistent states
✓JSONB поля — audit_log хранит payload (состояние до/после) в JSONB с индексацией. Гибкость NoSQL при надёжности SQL
✓Partial indexes, составные индексы — можно точечно оптимизировать медленные запросы без изменения схемы
✓PostGIS-совместимость — тип POINT для геолокации, в будущем можно добавить geo-запросы (заявки рядом с объектом)
✓Зрелость и надёжность — 30+ лет разработки, используется Shopify, Instagram, GitLab. Community и документация — лучшие в индустрии
Альтернатива: MySQL — нет полной поддержки JSONB, слабее partial indexes. MongoDB — нет ACID по умолчанию, сложнее JOIN-запросы для аналитики.
Redis Cache + Queue Store
✓In-memory кеш — аналитические запросы к PG занимают 200-400мс, с Redis кешем — 5-10мс. Разница в 40х для дашборда
✓BullMQ backend — SLA-таймеры, delayed jobs и очереди уведомлений хранятся в Redis. Надёжнее in-memory, переживает перезапуск сервера
✓Rate limiting — Redis INCR + EXPIRE для throttle: атомарный счётчик запросов без race condition
✓Session store — хранение refresh-токенов с автоматическим TTL. При logout токен просто удаляется из Redis
Альтернатива: Memcached — нет поддержки структур данных (List, Set, SortedSet) нужных BullMQ. RabbitMQ — сложнее в настройке, избыточен для данного масштаба.
React + AntD Frontend Stack
✓Ant Design — enterprise UI-библиотека от Alibaba. Содержит Table, Form, DatePicker, Charts, Drawer — всё что нужно для Admin панели, без кастомной верстки
✓React Query — server state management с кешированием, background refetch, оптимистичными обновлениями. Заменяет Redux для серверных данных
✓Zustand — минималистичный стейт для UI (фильтры, модалки, пользователь). Нет boilerplate как в Redux
✓Vite — сборка за 8-12 секунд (webpack занял бы 60+), HMR мгновенный. Developer Experience кратно лучше
Альтернатива: Next.js — SSR избыточен для Admin SPA. Vue + Element Plus — меньше специалистов на рынке. Angular — высокий порог входа, медленнее для прототипирования.
BullMQ Job Queue
✓Delayed jobs — SLA-таймер: создать job с задержкой 4 часа, который сработает и отправит эскалацию. Невозможно с обычным setTimeout
✓Retry с backoff — если Telegram API упал, уведомление повторится через 5с, 30с, 5мин. Гарантированная доставка
✓Concurrency control — параллельная обработка N jobs без перегрузки внешних API (rate limit на Telegram: 30 сообщений/сек)
✓Job events — NestJS слушает completed/failed события и обновляет БД или логирует результат
Альтернатива: node-cron — нет delayed jobs, нет retry, нет persistent storage. Agenda — использует MongoDB вместо Redis, дополнительная зависимость.
Telegram Bot API Mobile / Notifications
✓Zero installation — пользователям не нужно скачивать отдельное приложение. Telegram уже установлен у 95%+ сотрудников
✓Telegram Mini App — полноценный React SPA внутри Telegram. Работает на iOS и Android одновременно без App Store
✓Push-уведомления бесплатно — Telegram Push без FCM/APNs, сертификатов и настроек. Просто sendMessage
✓Авторизация через initData — безопасная идентификация пользователя без отдельного login flow в мобильном интерфейсе
Альтернатива: React Native — требует отдельной разработки iOS + Android, App Store review. PWA — нет push на iOS без браузера. Telegram покрывает 90% use cases быстрее и дешевле.

Безопасность

JWT, RBAC, шифрование и защита от уязвимостей

🔒
JWT Auth
Access token (15 мин) + Refresh token (30 дней). Хранение RT в httpOnly cookie. Rotation при обновлении.
👥
RBAC
Декоратор @Roles() на каждом endpoint. Guard проверяет роль из JWT payload перед выполнением.
💪
Bcrypt
Пароли хешируются bcrypt с salt rounds = 12. В БД хранится только хеш, не открытый пароль.
🛡
Rate Limiting
Throttle Guard: 100 req/min для обычных, 10 req/min для auth endpoints. Redis-based counter.
🔍
Валидация
class-validator + class-transformer на всех DTO. Whitelist-режим: лишние поля отклоняются.
🖥
CORS / Helmet
Список разрешённых origins в конфиге. Helmet.js устанавливает security-заголовки (CSP, HSTS, X-Frame).

Ресурсная изоляция данных

⚠
Принцип минимальных привилегий
Заказчик видит только свои заявки (WHERE customer_id = req.user.id). Исполнитель — только назначенные ему заявки. Менеджер — все заявки своей организации. Фильтрация происходит на уровне сервисного слоя, не в контроллере.

Аудит и логирование

Полный журнал действий и система структурированного логирования

📄
Audit Log

Каждое действие пользователя записывается в таблицу audit_logs:

  • Кто совершил действие (user_id, email, role)
  • Что сделал (action: CREATE | UPDATE | DELETE | LOGIN)
  • Над чем (entity: requests, users, categories…)
  • Payload: снепшот состояния до и после
  • IP-адрес и User-Agent
  • Timestamp с точностью до миллисекунды
🔧
Application Logging

Структурированное логирование через Winston:

  • Уровни: error, warn, info, debug
  • Формат: JSON (для агрегаторов)
  • Транспорты: console + file rotation
  • Correlation ID (X-Request-ID) в каждом запросе
  • Middleware логирует все HTTP запросы (метод, путь, статус, время)
  • Sentry для error tracking и alerting
